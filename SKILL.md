# GitHub Pull Request

## Vision

The goal of this skill is to formalize the process of proposing code changes for review. A well-formed pull request (PR) is the cornerstone of collaborative development and a critical control point for maintaining code quality. This skill ensures that every proposed change is documented, contextualized, and ready for human review.

## When to Use

Use this skill after you have committed a set of changes to a feature or bugfix branch and pushed it to the remote repository. It is the final step before submitting your work for review and potential inclusion in the `main` branch.

This skill is also the final step for automated processes like `automated-vulnerability-patching`.

## How to Create a Pull Request

1. **Ensure your branch is up to date:** Before creating a PR, make sure your branch is pushed to the remote repository (`git push`).
2. **Use the `gh` CLI:** The GitHub CLI (`gh`) is the preferred tool for this task.
3. **Construct the command:**

    ```bash
    gh pr create --title "feat: A brief, descriptive title" --body-file /path/to/pr_body.md --assignee "@me"
    ```

    - `--title`: A concise summary of the change. Use conventional commit prefixes (e.g., `feat:`, `fix:`, `docs:`, `chore:`).
    - `--body-file`: A path to a temporary markdown file containing the detailed description of the PR. This is much more robust than passing a long string to `--body`.
    - `--assignee`: The GitHub handle of the person who should review the PR.

## PR Description Templates

Using a template for the PR body ensures all necessary information is included.

### Standard Change Template

For typical feature or bugfix work, the body should include:

```markdown
## What does this PR do?

A clear and concise description of the changes.

## Why is this change needed?

The motivation behind the change. Link to any relevant issues (e.g., "Fixes #123").

## How to test

Step-by-step instructions for how a reviewer can verify the changes.
```
